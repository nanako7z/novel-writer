# Phase 01: Radar（市场雷达）

> ⛔ **硬约束 / 不跳步**：
> 1. **前置**：用户已给题材 / 平台 / 字数指标等市场信号；这是侧流，**不进**主循环、**不**写真理文件
> 2. **本阶段必跑**：5 维 radar 分析（题材热度 / 平台对位 / 字数节奏 / 同题材竞品 / 切入点）一项不漏
> 3. **退出条件**：答用户，**不写盘**；用户若要立项，转 SKILL.md "项目初始化"小节调 `init_book.py`
> 4. **重试规则**：n/a（无 LLM 解析闸门）

## 何时进入

- v1 中**默认不进**主循环。inkos 原本会用 `inkos radar` 抓榜单 + LLM 分析市场，本 SKILL 形态做不到实时抓取。
- 仅当**用户明确要求**或**项目初始化时还没确定题材**才进入。
- 入口语句示例：
  - "现在玄幻什么火？"
  - "我想做一本爽文但不知道选什么细分赛道"
  - "番茄最近哪种开局收藏好？"

## Inputs

- 用户提供的市场信号（必需），来源可以是：
  - 用户口述的"番茄玄幻榜前 10 名是 XXXX"
  - 用户粘贴的榜单截图 / txt
  - 用户给的链接（Claude Code 可调 web fetch 时可拉一次）
- 题材范围（可选）：用户限定"玄幻 / 都市 / 奇幻"等
- 平台（可选）：番茄 / 飞卢 / 起点 / 其他

## Process

### 系统 prompt（搬自 inkos `agents/radar.ts`，本 SKILL 改为静态注入）

```
你是一个专业的网络小说市场分析师。下面是从各平台实时抓取的排行榜数据，请基于这些真实数据分析市场趋势。

分析维度：
1. 哪些题材在涨（与上周/上月对比；如无对比数据则基于绝对量级与用户给的描述判断）
2. 共同的开局套路（黄金 3 章普遍写什么）
3. 高收藏 / 低差评的作品有什么共性
4. 哪些细分赛道有空缺、值得切入

输出格式必须为 JSON：
{
  "recommendations": [
    {
      "platform": "tomato | feilu | qidian | other",
      "genre": "<细分赛道>",
      "concept": "<一句话描述切入点>",
      "confidence": 0.0-1.0,
      "reasoning": "<为什么这个赛道现在能做>",
      "benchmarkTitles": ["对标作 1", "对标作 2"]
    }
  ],
  "marketSummary": "整体市场概述（2-3 句）"
}
```

### 工作步骤

1. **拿不到一手榜单数据时直接退出**——不要凭印象编榜单。告诉用户："请把番茄玄幻榜前 10 名（含书名+简介+收藏数）粘给我。"
2. 拿到数据后，按上面 prompt 在心里走一遍，不实际让用户看 JSON——用户更想要的是**人话推荐**。
3. 给出 2-3 个具体切入点，每个：题材 + 一句话定位 + 为什么现在能做 + 一两本对标。
4. 如果用户说"行，就做这个"，把选定项的 `genre` / `platform` 填到 `book.json`，进入 init 流。

## Output contract

- 不写文件。
- 输出形式：直接对话回复用户。
- 若用户决定立项，把 `genre` / `platform` / 简短的题材描述（写到 `templates/story/author_intent.md` 的占位里）作为入参传给 `scripts/init_book.py`。

## Failure handling

- 用户没给数据 → 不要瞎编。要求补充。
- 数据不一致（书名收藏数对不上） → 标出怀疑点，让用户复核，不强行分析。
- 用户给的题材限制太死（"我只写修仙脑洞"）→ 不必强推榜单趋势，按用户既定方向给 1 个推荐 + 1 个略微偏离的对照即可。

## 注意事项

- **本 phase 不是写作主循环的一部分**。它产出的"题材+定位"是 init 流的输入，不是 plan / write 的输入。
- inkos 的 daemon 模式（`inkos up` 自动每 6 小时跑一次 radar）不在 SKILL 形态内复刻。要做定时趋势复盘，用 `/schedule` 起一个 routine。
- 网络抓取走 web fetch / MCP 时风险较大（榜单页面常有反爬），优先让用户主动复制粘贴。
