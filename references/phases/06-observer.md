# Phase 06: Observer（事实抽取）

> ⛔ **硬约束 / 不跳步**：
> 1. **前置**：本章已通过 audit-revise 闭环（step 7 退出）的 `draft`（最终 normalized 稿）就位；**禁止**用 raw_writer 原文跳过 audit 直接喂 Observer
> 2. **本阶段必跑**：OBSERVATIONS 块覆盖 9 类事实（角色 / 资源 / 关系 / 钩子兑现 / 钩子新增 / 场景 / 时间线 / 情绪 / 重要细节）一类不漏——只记正文真发生的，**禁止**从大纲推断
> 3. **退出条件**：`story/runtime/observations.md` 落盘后才能进 step 9 Settler
> 4. **重试规则**：OBSERVATIONS 解析失败重试 ≤ 1 次，注入"上次缺哪一类"的具体反馈；仍失败降级——跳过 observations，由 Settler 从原文抽取

## 何时进入

主循环 step 8，Writer 之后、Settler 之前。从正文**穷举**可观察事实变化（9 类），输出 `=== OBSERVATIONS ===` 块给 Settler 增量更新真理文件。Observer 自身不写真理文件。

**宁多勿少**：Observer 过度抽取，由 Settler 做合并过滤——这是本阶段的设计原则。

## Inputs

Claude 在这一阶段需要读：

- 刚写完的章节正文（`chapters/<NNNN>.md`）+ 章节标题、章节号
- `book.json` ——title / language / genre
- `references/genre-profiles/<genre>.md` ——题材语境（"打怪"是动作还是情绪？某些"得到"在数值题材里要记账）

不读真理文件——Observer 是"只看正文"的角色，避免把上一章的旧事实带进来。

## Process

Claude 在心中扮演"事实提取专家"，按下面的系统 prompt 执行。**只从正文提取，不推测可能发生的事。**

### 系统 prompt

```
你是一个事实提取专家。阅读章节正文，提取每一个可观察到的事实变化。

## 提取类别

1. **角色行为**：谁做了什么，对谁，为什么
2. **位置变化**：谁去了哪里，从哪里来
3. **资源变化**：获得、失去、消耗了什么，具体数量
4. **关系变化**：新相遇、信任/不信任转变、结盟、背叛
5. **情绪变化**：角色情绪从X到Y，触发事件是什么
6. **信息流动**：谁知道了什么新信息，谁仍然不知情
7. **剧情线索**：新埋下的悬念、已有线索的推进、线索的解答
8. **时间推进**：过了多少时间，提到的时间标记
9. **身体状态**：受伤、恢复、疲劳、战力变化

## 规则

- 只从正文提取——不推测可能发生的事
- 宁多勿少：不确定是否重要时也要记录
- 具体化："陆承烬左肩旧伤开裂" 而非 "陆承烬受伤了"
- 记录章节内的时间标记
- 标注每个场景中在场的角色

## 输出格式

=== OBSERVATIONS ===

[角色行为]
- <角色名>: <行为/状态变化> (场景: <地点>)

[位置变化]
- <角色> 从 <A> 到 <B>

[资源变化]
- <角色> 获得/失去 <物品> (数量: <n>)

[关系变化]
- <角色A> → <角色B>: <变化描述>

[情绪变化]
- <角色>: <之前> → <之后> (触发: <事件>)

[信息流动]
- <角色> 得知: <事实> (来源: <途径>)
- <角色> 仍不知: <事实>

[剧情线索]
- 新埋: <描述>
- 推进: <已有线索> — <进展>
- 回收: <线索> — <解答>

[时间]
- <时间标记、时长>

[身体状态]
- <角色>: <受伤/恢复/疲劳/战力变化>
```

### 用户消息模板

```
请提取第{chapterNumber}章「{title}」中的所有事实：

{content}
```

英文书改写为：

```
Extract all facts from Chapter {chapterNumber} "{title}":

{content}
```

英文版 9 类标签直接沿用 inkos `observer-prompts.ts` 的 isEnglish 分支（`[CHARACTERS]` / `[LOCATIONS]` / `[RESOURCES]` / `[RELATIONSHIPS]` / `[EMOTIONS]` / `[INFORMATION]` / `[PLOT_THREADS]` / `[TIME]` / `[PHYSICAL_STATE]`）。

### 工作步骤

1. **读正文与上下文**。只读章节正文 + book/genre 的语言和题材；**不读真理文件**。
2. **逐场景扫**。按场景顺序读，每一段心里维护一份"在场角色 + 当前位置"。
3. **9 类全跑**。即使某类本章无事，也输出 `[<类别>]` 标题与一行"（本章无）"，保证下游解析稳定。
4. **拒绝推断**。只写正文里**写出来的**事——主角"走到门口"时不要写"主角进入房间"；某事被暗示但未确认时不要当作既成事实。
5. **数值具体化**。题材有数值系统（修仙 / 系统 / 游戏）时，资源变化必须写出具体数量（"获得灵石 30 块"而非"得到灵石若干"）。
6. **信息边界**。`[信息流动]` 是 Settler 跑角色交互矩阵的输入——必须标"谁知道 / 谁仍不知"两侧；只在场的角色才知道在场时发生的事。
7. **新线索 vs 推进**。`[剧情线索]` 区分三种状态：`新埋`（本章首次出现）/ `推进`（已有线索新增事实、证据、风险升级）/ `回收`（明确解答或不再成立）。**重述、抽象总结、换说法不是推进**。
8. **输出 `=== OBSERVATIONS ===`** 一整块，不要包代码块标记，不要加多余说明。

## Output contract

- 写入 `story/runtime/observations.md`（覆盖式）
- 单一文本块，以 `=== OBSERVATIONS ===` 起首
- 9 个 `[类别]` 子块按固定顺序出现（中文：角色行为 / 位置变化 / 资源变化 / 关系变化 / 情绪变化 / 信息流动 / 剧情线索 / 时间 / 身体状态）
- 子块下用 `- ` 列表项；本章无对应事实写一条 `- （本章无）`
- 该文件由 Settler 直接读取作为 user 消息的 observations 块

不需要 JSON schema——纯 markdown 块。

## Failure handling

Observer 不会"解析失败"——输出是自由格式 markdown 块。但有以下约束：

- **9 类标签必须齐全**：缺某个标签 → Settler 会报"observations 不完整"，主循环重跑 Observer 一次；第 2 次仍缺标签 → 抛错给用户。
- **标签外内容**：如果 Claude 在 9 个 `[类别]` 之外写了"分析"、"总结"、"建议"等内容，Settler 会忽略这些行；不影响主流程，但下次 Observer 应避免。
- **正文过长**：当章超过 1.5x 目标字数时，Observer 可能漏抓——Settler 接收到不完整 observations 时会做侧检（与上一章对比），但不强制 Observer 重跑。

最大重试 1 次；超过仍失败 → 跳过 observations 直接进 Settler，由 Settler 从原文抽取（降级路径）。

## 注意事项

- **过度抽取是美德**：Observer 是 Settler 的输入端，宁多勿少。Settler 自己会过滤"重述 / 换说法"那些非真实推进，但不能凭空生出 Observer 没看到的事实。
- **不写真理文件**：Observer 的产物是临时观察日志（`story/runtime/observations.md`），下一章会被覆盖。真理文件由 Settler 写。
- **不做合并**：Observer 不读旧的 hooks.json / current_state.json，不要试图判断"这是新事实还是已有事实"——那是 Settler 的活。
- **数值在数值题材里必填**：如果题材的 `numericalSystem === true`，资源/数值变化没写具体数字 → Auditor 第 2 维度（资源账本）会爆。
- **场景标注**：`[角色行为]` 末尾的 `(场景: <地点>)` 不是装饰——Settler 用它来推断信息边界（哪些角色"在场"）。
- **English book**：直接用 inkos 英文版 9 个标签；输出语言完全跟随正文语言。
- **新埋 hook 候选**：Observer 写 `[剧情线索] - 新埋: <描述>` 时不要发明 hookId——hookId 由 Settler 通过 `newHookCandidates` 路径决定。
